Dataverse as a State Machine:
Approval Workflows That Scale

Stop putting your approval logic in Power Automate flows. Here's the pattern that separates systems that last from ones that quietly break at 3am.

I've reviewed a lot of Power Platform solutions across a lot of organizations. And almost every approval system I've encountered shares the same original sin: the business logic lives in the flow.

I understand why it happens. Power Automate is where you go when you think "automation." It's visual, it's fast, and it feels like the right place to say "when X happens, do Y, unless Z, then do W." So you build a flow. Then you add conditions. Then you add more conditions. Then someone asks for a new approval tier, and you realize you're now maintaining a 300-action flow that nobody — including you — can fully reason about anymore.

There's a better way. And it starts with a shift in how you think about what Dataverse actually is.

The Problem with Flow-First Logic

Let me describe a scenario you've probably seen. A company needs a customer account approval workflow. Accounts go through three stages: Submitted → Manager Review → Finance Review → Active. Someone builds a flow. It works. Six months later, requirements change:

  • Finance review is now optional, based on account value
  • Rejected accounts can be resubmitted, but only twice
  • If the submitter is also the manager, it auto-escalates
  • Any stage can be put on Hold by a supervisor

The flow now has 47 conditions, four parallel branches, and three different trigger points. When it breaks — and it will — you get to spend an afternoon reading a 400-row run history trying to find which branch silently swallowed an error.

⚠
The real cost isn't building the flow. It's every future change, every debugging session, every "why did this account get stuck?" conversation that follows. Flow-first approval logic doesn't scale with business complexity — it multiplies it.

The root issue is that flows are stateless executors. They run, they finish. They're not designed to be the state — they're designed to react to it. The moment you start embedding state in your flow logic, you've inverted the architecture.

The Core Insight

Here's the reframe that changes everything:

Dataverse is your state machine. Power Automate is just a notification bus.

Every approval workflow is fundamentally a finite state machine. There are a fixed number of possible states (Submitted, Under Review, Approved, Rejected…), and a fixed set of transitions between them (Submit → Review, Approve → Active, Reject → Draft…). Each transition has conditions that must be true before it's valid.

When you model this in Dataverse — with proper Choice columns for status, Business Rules or plugins for transition validation, and Dataverse's native audit log — your flow becomes trivially simple. It just watches for state changes and sends emails. That's it. The logic is in the data model, where it belongs.

Fig 1 — Flow-First vs State-First architecture

Designing the State Machine

Before touching Dataverse, you need to map your states and transitions on paper. This is the most important step and the one most teams skip. A state machine has three things:

  • States — the finite set of situations a record can be in
  • Transitions — the valid moves between states, and who can trigger them
  • Guards — conditions that must be true for a transition to be allowed

The statuscode column is the crown jewel. It's a Choice column with every valid state as an option. But a Choice column alone is just a dropdown — what makes it a state machine is the plugin that guards every write to it.

Transition Rules & Validation

The plugin runs on PreValidation of the Update message, targeting statuscode. Before any status change is committed to the database, the plugin checks a transition matrix — a dictionary that maps every valid (fromState, toState) pair to its guard conditions.

C# — Plugin: ValidateStatusTransition 
public class ValidateStatusTransition : IPlugin
{
    // Define valid transitions: (from, to) → guard condition key
    private static readonly Dictionary<(int from, int to), string> _validTransitions = new()
    {
        { (Draft,     Submitted),       "CanSubmit"       },
        { (Submitted, ManagerReview),   "CanAssign"       },
        { (ManagerReview, Active),      "CanApprove"      },
        { (ManagerReview, FinanceReview),"NeedsFinance"   },
        { (ManagerReview, Rejected),    "CanReject"       },
        { (ManagerReview, OnHold),      "CanHold"         },
        { (FinanceReview, Active),      "CanApprove"      },
        { (FinanceReview, Rejected),    "CanReject"       },
        { (Rejected,  Draft),           "CanResubmit"     },
        { (OnHold,    ManagerReview),   "CanResume"       },
    };

    public void Execute(IServiceProvider serviceProvider)
    {
        var context = (IPluginExecutionContext)serviceProvider
            .GetService(typeof(IPluginExecutionContext));
        var target = (Entity)context.InputParameters["Target"];

        if (!target.Contains("statuscode")) return;

        var orgService = /* ... get service ... */;
        var current = orgService.Retrieve("khatib_customeraccount",
            context.PrimaryEntityId, new ColumnSet("statuscode", 
            "khatib_rejectioncount", "khatib_submittedby",
            "khatib_accountvalue", "ownerid"));

        int fromState = current.GetAttributeValue<OptionSetValue>("statuscode").Value;
        int toState   = target.GetAttributeValue<OptionSetValue>("statuscode").Value;

        // Is this a valid transition at all?
        if (!_validTransitions.TryGetValue((fromState, toState), out var guardKey))
            throw new InvalidPluginExecutionException(
                $"Invalid status transition: {fromState} → {toState}");

        // Evaluate the guard condition
        if (!EvaluateGuard(guardKey, current, context))
            throw new InvalidPluginExecutionException(
                GetGuardFailureMessage(guardKey, current));

        // Write audit history before the change commits
        WriteStatusHistory(orgService, context.PrimaryEntityId,
            fromState, toState, context.InitiatingUserId);
    }

    private bool EvaluateGuard(string guardKey, Entity record, 
        IPluginExecutionContext context)
    {
        return guardKey switch
        {
            "CanResubmit"  => record.GetAttributeValue<int>(
                                  "khatib_rejectioncount") < 2,
            "NeedsFinance" => record.GetAttributeValue<Money>(
                                  "khatib_accountvalue")?.Value > 50_000,
            "CanApprove"   => IsAuthorizedReviewer(record, context),
            "CanReject"    => IsAuthorizedReviewer(record, context),
            "CanHold"      => IsSystemAdmin(context),
            _              => true   // no additional guard
        };
    }
}

A few things worth calling out here:

  • The transition matrix is the entire business logic. Adding a new rule means adding one entry to the dictionary and one method. No flow to touch.
  • PreValidation means nothing commits if the guard fails. No partial saves, no corrupt state, no cleanup flows needed.
  • The audit write happens in the same transaction. You get atomicity — the status change and its history record either both succeed or both roll back.
💡
Use a Custom API instead of a direct update if you want to expose the transition as a business operation (e.g., khatib_SubmitAccount, khatib_ApproveAccount). This gives you named, documented actions that Canvas Apps and flows can call without needing to know the underlying status values. It also lets you pass additional parameters — like a rejection reason — cleanly alongside the state change.

Audit Trail That Actually Works

Because WriteStatusHistory is called inside the plugin — in the same database transaction as the status change — you get a complete, tamper-proof audit trail with zero extra effort from the UI layer. Every record in khatib_statushistory tells you:

  • What the status was before
  • What it changed to
  • Who changed it (the initiating user, not the integration user)
  • When it happened (database timestamp, not client time)
⚠
Don't use Dataverse's built-in audit log for this. The native audit log is not easily queryable in Canvas Apps, it has storage implications in managed environments, and it doesn't let you store custom context (like the reviewer's comments). Your own status history table is cleaner, faster, and completely under your control.

Real-World: Multi-Tier Approval

On the Masheed project, we implemented a customer account approval with three approval tiers, a consensus mechanism at Tier 2 (all Tier 2 approvers had to agree, not just one), and a Canvas App for approvers to view and act on pending requests. Here's how the Canvas App side looked — radically simple because the complexity lived in Dataverse, not the app:

Power Fx — Approve button OnSelect 
// This is the ENTIRE approval button logic.
// No status values. No condition checking. No if/else chains.
// The Custom API does all of that — and throws a clear error if it fails.

Set(varIsLoading, true);

With(
    { result: khatib_ApproveAccount.Run(
        galPendingAccounts.Selected.khatib_customeraccountid,
        txtComments.Text
      )
    },
    If(
        result.success,
        Notify("Account approved successfully", NotificationType.Success);
        Refresh(khatib_customeraccount),
        Notify("Could not approve: " & result.errormessage, NotificationType.Error)
    )
);

Set(varIsLoading, false);

The Canvas App doesn't need to know about states, counts, or guard conditions. It calls a named action, handles success or failure, and moves on. This is what it looks like when the separation of concerns is right.

Gotchas & Edge Cases

A few things that will bite you if you're not careful:

01
System updates bypass your plugin. If a flow or another plugin updates the status using the system user context (e.g., with ExecuteAsSystemUser), your validation is skipped. Always use the initiating user context, and make sure your integration flows run as named users, not the integration user.
02
Bulk operations are tricky. When someone bulk-updates 50 accounts, each one will hit your plugin individually — which is correct — but your validation error messages need to be per-record, not per-batch. Test this scenario explicitly.
03
Status reason vs status code. Dataverse has both statecode (Active/Inactive) and statuscode (Status Reason — your custom states). Use statuscode for your workflow states. Don't conflate the two. And resist the temptation to use Inactive state as "Rejected" — it prevents the record from being updated at all, breaking your resubmission path.
04
Error messages are your documentation. When a guard fails, the exception message is what the user sees. Write them in plain language: "This account has been rejected twice and cannot be resubmitted" is infinitely better than "Guard condition 'CanResubmit' evaluated to false."

Key Takeaways

◈

State belongs in Dataverse

Your Choice column, your transition matrix, your guard plugin — this is where business logic lives. Not in flows.

◈

Flows are notification buses

One trigger, one switch, one email per case. If your flow has conditions about business rules, something is wrong.

◈

Custom APIs are your surface

Named actions (ApproveAccount, RejectAccount) decouple your UI from your data model and make the system self-documenting.

◈

PreValidation for atomicity

History records written in the same transaction as the status change. No orphaned records, no partial state.

◈

Map first, build second

Draw your state machine on paper before touching Dataverse. Every hour spent there saves three in debugging.

◈

Your audit table beats native logs

Own your history. Query it in Canvas Apps, export it, report on it. Don't depend on platform internals.

This pattern — Dataverse as state machine, flow as orchestrator — is the single biggest architectural upgrade I've applied to approval workflows in the Microsoft ecosystem. Once you build one this way, you won't go back.
