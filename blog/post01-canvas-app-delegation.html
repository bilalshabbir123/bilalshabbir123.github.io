Delegation in Canvas Apps: Stop Hitting the 500-Row Limit

Bilal Shabbir May 2026 8 min read

Your gallery looks perfect in development. Then you deploy to production and suddenly half the records are missing — with no error message explaining why. Delegation is the most misunderstood concept in Canvas Apps, and it silently breaks apps that work fine with small datasets. Here's how it actually works and how to design around it.

What is delegation?

When you write a Power Apps formula against a data source like SharePoint or Dataverse, Power Apps has two choices: run the query locally (pull all records to the client, then filter) or delegate the query to the server (let the data source do the filtering). Delegation means the second option.

The problem is that local processing has a hard cap — Power Apps will only pull a maximum of 2,000 records by default (you can raise this to 2,000 in settings, but that's still the ceiling). If your SharePoint list has 10,000 items and you filter locally, you're silently operating on only the first 2,000 items. The yellow delegation warning triangle in the formula bar is trying to tell you this.

The silent failure problem: Power Apps does not show an error when delegation fails. Your gallery will render, show items, and appear to work correctly. Users will never know they're seeing incomplete results.

Which functions delegate and which don't

Delegation support varies by connector. Here's what you need to know for the two most common data sources:

SharePoint Online — what delegates

  • Filter() with =, <>, <, >, <=, >= on most column types
  • StartsWith() on text columns
  • Search() — but only on indexed columns
  • Sort() and SortByColumns()
  • CountRows() when used with a delegable filter

SharePoint Online — what does NOT delegate

  • In operator ("value" in column)
  • Lower(), Upper(), Len(), Left(), Right(), Mid()
  • IsBlank() in a filter
  • DateDiff(), DateAdd()
  • Nested Filter() calls
  • Any formula on a Person/Group or Lookup column beyond basic equality
Dataverse delegates far more than SharePoint. If you're building on Dataverse you get full delegation for Filter, LookUp, string functions, date comparisons, and complex boolean logic. SharePoint's delegation surface is much smaller — keep this in mind when choosing your data source for large datasets.

Pattern 1 — Move heavy filtering server-side

The cleanest fix is to only use delegable predicates in your Filter() call. Instead of filtering on a calculated or formula-derived value, store the value you need directly in the list. For example, instead of filtering by "items where DueDate is in the past" using DateDiff, add an indexed IsOverdue boolean column that gets set by a Power Automate flow, and filter on that. 

// ❌ Not delegable — DateDiff doesn't push to server
Filter(
  ProjectTasks,
  DateDiff(Today(), DueDate, Days) < 0
)

// ✅ Delegable — filtering on a stored boolean
Filter(
  ProjectTasks,
  IsOverdue = true
)

Pattern 2 — Collections for small-but-static reference data

If your dataset is small (under 2,000 rows) and relatively stable — like a lookup table, status codes, department list — load it all into a collection on OnStart or the screen's OnVisible. Then filter the collection locally. Collections are always local so delegation doesn't apply, and the data is already in memory so filtering is instant. 

// OnStart or OnVisible
ClearCollect(
  colDepartments,
  Departments  // SharePoint list with <500 rows
);

// Gallery Items — fast local filter, no delegation concern
Filter(
  colDepartments,
  Region = varSelectedRegion
)

Pattern 3 — Named Formulas for shared computed values

Named Formulas (in Power Apps Studio → App → Formulas) let you define computed values once and reuse them throughout the app. Unlike variables, they recalculate automatically when their dependencies change — so you avoid duplicating filtering logic across screens. 

// App.Formulas
ActiveTickets = Filter(
  SupportTickets,
  Status = "Open",
  AssignedTo.Email = User().Email
);

Now ActiveTickets is available on every screen and recalculates reactively. Because the underlying Filter uses a delegable predicate, this scales to any number of records.

Pattern 4 — Pagination for large result sets

When users genuinely need to browse a large dataset, implement pagination instead of trying to load everything. Use the Last() / First() pattern with Filter and a tracked page cursor: 

// On next-page button press
Set(
  varPageCursor,
  Last(Gallery1.AllItems).ID
);

// Gallery Items — filter records after the cursor
Filter(
  SupportTickets,
  Status = varFilterStatus,
  ID > varPageCursor
)

How to audit your app for delegation issues

Power Apps Studio shows a yellow warning triangle on any formula with delegation issues. But these are easy to miss, especially in complex apps. Do this instead:

  1. Go to File → App Settings → Advanced Settings
  2. Set Data row limit to 5 temporarily
  3. Test your app — any gallery that should show more than 5 items but shows exactly 5 (or fewer) has a delegation problem
  4. Reset to 500 or 2000 once you've identified all problem areas
Don't just raise the limit and call it done. The 2,000-row cap is the ceiling, not a solution. Raising it from 500 to 2,000 just hides the problem for longer. If your data source has more than ~1,500 rows of data relevant to any query, fix the delegation issue properly.

Summary

  • Delegation means the data source does the filtering, not the client
  • Non-delegable formulas silently cap results at 500–2,000 rows with no error
  • Prefer delegable predicates (=, StartsWith, indexed column comparisons) in Filter()
  • Use collections for small, stable reference data
  • Use Named Formulas for shared filtering logic
  • Test with a row limit of 5 to surface any delegation gaps quickly
