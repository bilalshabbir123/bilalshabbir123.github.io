Error Handling in Power Automate That Actually Works

Bilal Shabbir May 2026 7 min read

Most Power Automate flows fail silently. The run shows a red cross, no one gets notified, and the business process quietly breaks. This guide covers the four error handling patterns I use on every production flow — from Run After settings to parallel error branches and Teams alert cards that tell you exactly what went wrong.

Why default error handling is not enough

Out of the box, a failed action stops the flow and marks the run as failed. Unless someone is actively monitoring the flow's run history, the failure goes unnoticed. For anything that matters — document processing, finance alerts, customer notifications — that's not acceptable.

Good error handling in Power Automate means three things:

  • Capturing what failed and why
  • Notifying the right person immediately
  • Ensuring partial completions don't leave data in a broken state

Pattern 1 — Run After settings

Every action in a flow has a Run After configuration. By default, an action only runs if all previous actions succeeded. You can override this to run an action after failed, skipped, or timed out as well.

The simplest error handler is a "Send email" or "Post to Teams" action configured to run after a specific step has failed:

  1. Click the three dots on any action → Configure run after
  2. Uncheck "is successful", check "has failed"
  3. Add a Teams notification action with the error details
Limitation: Run After on individual actions only handles that specific action's failure. If you have 20 actions and any one can fail, you'd need 20 error handlers. Use Scope actions instead (see below).

Pattern 2 — Scope actions for structured try/catch

A Scope action is Power Automate's equivalent of a try/catch block. It groups related actions, and you can configure a single handler to run if any action inside the scope fails.

The structure looks like this:

Scope: Try
  ├─ Action 1
  ├─ Action 2
  └─ Action N

Scope: Catch  (Run After: Try has failed)
  ├─ Get error details
  └─ Post Teams message

Inside the Catch scope, use the result() function to get error details from the Try scope:

// In the Teams message body — paste this as an expression
concat(
  'Flow failed: ',
  result('Try')[0]?['error']?['message']
)

This gives you the actual error message rather than just "something went wrong".

Pattern 3 — Storing full error context

The result() function returns a full JSON object for each action in the scope — status, start/end time, inputs, outputs, and error. For production flows I always store this to a SharePoint list or SQL table before sending the notification, so there's a permanent audit trail. 

// Initialize variable in Try scope
Initialize variable
  Name: varRunId
  Value: @{workflow()['run']['name']}

// In Catch scope — create SharePoint error log entry
Create item — FlowErrorLog list
  FlowName: Document Processor
  RunId: @{variables('varRunId')}
  ErrorMessage: @{result('Try')[0]?['error']?['message']}
  FailedAction: @{result('Try')[0]?['name']}
  Timestamp: @{utcNow()}

Pattern 4 — Teams Adaptive Card alerts

Plain text notifications get ignored. An Adaptive Card with the error details, a link to the run history, and a "Re-run" button gets acted on. Here's the card body I use: 

{
  "type": "AdaptiveCard",
  "$schema": "http://adaptivecards.io/schemas/adaptive-card.json",
  "version": "1.4",
  "body": [
    {
      "type": "TextBlock",
      "text": "⚠️ Flow Failed",
      "size": "Large",
      "weight": "Bolder",
      "color": "Attention"
    },
    {
      "type": "FactSet",
      "facts": [
        { "title": "Flow", "value": "Document Processor" },
        { "title": "Error", "value": "@{result('Try')[0]?['error']?['message']}" },
        { "title": "Time", "value": "@{formatDateTime(utcNow(), 'dd MMM yyyy HH:mm')}" }
      ]
    }
  ],
  "actions": [
    {
      "type": "Action.OpenUrl",
      "title": "View Run History",
      "url": "https://make.powerautomate.com/..."
    }
  ]
}

Handling timeouts separately

HTTP actions and some connectors can time out independently of failing. Configure your Catch scope's Run After to include timed out in addition to failed. For flows that call external APIs, add a timeout policy on the HTTP action itself (under Settings → Timeout) so you control when it gives up rather than waiting for the platform's default.

The complete error handling structure

Every production flow I build follows this skeleton:

Trigger
│
├─ Scope: Validate Inputs
│   └─ Condition: required fields present?
│       └─ No → Terminate (Failed, "Missing required input: ...")
│
├─ Scope: Try
│   ├─ Main business logic actions
│   └─ Initialize varRunId = workflow()['run']['name']
│
└─ Scope: Catch  ← Run After: Try has failed OR timed out
    ├─ Create item in FlowErrorLog
    ├─ Post Adaptive Card to Teams channel
    └─ Terminate (Failed, error message)
Don't forget Terminate. If you handle the error but don't explicitly terminate the flow as Failed, the run shows as Succeeded — which is misleading. Always end your Catch scope with a Terminate action set to Failed.

Summary

  • Use Scope (Try/Catch) rather than per-action Run After — it scales
  • Use result('ScopeName')[0]?['error']?['message'] to get the actual error text
  • Log errors to SharePoint or SQL for a permanent audit trail
  • Send Teams Adaptive Cards with a link to the run history
  • Always terminate as Failed at the end of your Catch scope
