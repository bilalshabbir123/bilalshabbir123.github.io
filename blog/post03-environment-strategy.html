Power Platform Environment Strategy for Real Teams

Bilal Shabbir May 2026 9 min read

Most teams start Power Platform the same way: everyone builds in the default environment, nothing is in source control, and deploying to production means copy-pasting manually. That works until it doesn't — usually at the worst possible time. This is the environment strategy and deployment approach I use for teams beyond the trial stage.

Why the default environment is a trap

The default environment in Power Platform is shared across your entire tenant. Every user has maker rights by default. There's no governance, no ALM, and no separation between someone's experimental app and a business-critical process. The first time a maker deletes a shared flow that production depends on, you'll understand why isolation matters.

The three-environment baseline

For any solution that matters to the business, you need three environments: 

Development (Dev)
  ↓  Export managed solution
Test / UAT
  ↓  Export managed solution
Production (Prod)
  • Dev — makers work here. Unmanaged solutions. Multiple developers can work simultaneously on feature branches.
  • Test / UAT — managed solution imported from Dev. Used for QA and user acceptance. Business users sign off here before anything goes to Prod.
  • Production — managed solution imported from Test. Only the deployment pipeline has write access. No direct editing.
Managed vs Unmanaged solutions: Unmanaged solutions are editable — use them in Dev. Managed solutions are locked — use them in Test and Prod. Importing a managed solution to an environment where an unmanaged version exists will cause problems; never mix managed and unmanaged in the same environment.

Solution naming and structure

A solution is your deployment unit. Every component — app, flow, table, connection reference, environment variable — should be inside a solution. Never build directly in an environment without a solution; components built outside solutions can't be moved.

I follow this naming convention:

// Publisher prefix: 3-4 letters for your org
Publisher: contoso (prefix: cnt)

// Solution names: PascalCase, describe the business domain
cnt_SupportManagement_Core
cnt_SupportManagement_Extensions

// Version: match your release cycle
1.0.0.0  → initial release
1.0.0.1  → patch
1.1.0.0  → minor feature
2.0.0.0  → breaking change

Environment Variables — the right way to handle config

Environment variables let you change configuration (SharePoint URLs, SQL connection strings, feature flags) between environments without modifying the solution. Define them in your solution and set their values in each environment separately. 

// In your solution — define environment variables
SupportSiteUrl  (type: string, default: https://contoso.sharepoint.com/sites/support-dev)
MaxRetryCount   (type: integer, default: 3)
EnableAlerts    (type: boolean, default: false)

// In Test environment — override
SupportSiteUrl → https://contoso.sharepoint.com/sites/support-test

// In Prod environment — override
SupportSiteUrl → https://contoso.sharepoint.com/sites/support
EnableAlerts   → true
Never hardcode environment-specific values in flows, apps, or connection references. If a SharePoint URL is hardcoded in a flow, every deployment to a new environment requires manual editing — which defeats the purpose of managed solutions.

Connection References — not just environment variables

Connection references are how solutions handle connector credentials across environments. When you add a connector action to a flow inside a solution, Power Apps creates a connection reference automatically. Each environment then maps that reference to the appropriate service account credentials.

The key rule: use service accounts for all production connections. Never use a personal user account for a production flow's connections. If that employee leaves, the connection breaks.

Deploying with Azure DevOps Pipelines

Manual deployments don't scale and introduce human error. The Power Platform Build Tools extension for Azure DevOps gives you pipeline tasks for every deployment step: 

# azure-pipelines.yml — simplified deployment pipeline

stages:
  - stage: ExportFromDev
    jobs:
      - job: Export
        steps:
          - task: PowerPlatformExportSolution@2
            inputs:
              authenticationType: PowerPlatformSPN
              PowerPlatformSPN: dev-service-connection
              SolutionName: cnt_SupportManagement_Core
              SolutionOutputFile: $(Build.ArtifactStagingDirectory)/solution.zip
              Managed: true

  - stage: DeployToTest
    dependsOn: ExportFromDev
    jobs:
      - job: Import
        steps:
          - task: PowerPlatformImportSolution@2
            inputs:
              authenticationType: PowerPlatformSPN
              PowerPlatformSPN: test-service-connection
              SolutionInputFile: $(Pipeline.Workspace)/solution.zip

Common mistakes and how to avoid them

  • Building outside a solution: Components created directly in an environment (not inside a solution) cannot be exported. Always create your app/flow from within a solution in Power Apps Studio.
  • Using personal accounts for connections: Production flows using personal accounts break when the user leaves. Create service accounts and map connection references to them.
  • Skipping the Test environment: Going directly Dev → Prod seems faster until a bug slips through. Even a lightweight UAT sign-off catches problems before they affect real users.
  • Not versioning solutions: Increment the solution version on every deployment. This creates a deployment history and lets you roll back to a known state.
  • Sharing flows in Prod directly: In production, flows should run as the service account, not be shared with individual users for editing. Share the canvas app for user access; keep the flow ownership separate.

Summary

  • Use three environments: Dev (unmanaged), Test (managed), Prod (managed)
  • Everything lives inside a solution — no orphaned components
  • Environment variables and connection references handle all config differences
  • Service accounts own all production connections — never personal accounts
  • Automate deployments with Azure DevOps + Power Platform Build Tools
  • Version every deployment and keep a pipeline-driven history
